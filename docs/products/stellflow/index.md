# Stellflow · 彗流

<div class="product-logo">
  <img src="/logo/stellflow.png" alt="Stellflow Logo">
</div>

> 消息队列与事件流平台，负责异步解耦、流式分发、削峰填谷和事件驱动集成。

## 产品定位

### 目标定位

Stellflow 提供统一的消息投递与事件分发能力，支持业务异步化、系统集成和流式处理场景。

### 适用边界

- 面向异步解耦、事件驱动和数据分发场景
- 面向延迟消息、顺序消息和流式消费模型
- 不替代任务调度平台，而是聚焦消息持久化与分发

## 核心能力

### 能力清单

- 支持普通队列、主题订阅、延迟消息和顺序消息
- 支持消费组、重试队列、死信队列和回溯消费
- 支持事件桥接、流式处理和跨 Region 复制

### 设计价值

- 通过异步化降低系统耦合和峰值冲击
- 通过可靠投递与位点管理支撑事件驱动架构

## 概要设计

### 控制面设计

- 治理层负责主题配置、配额、审计和事件追踪
- 主题、消费组和复制策略由控制面统一管理

### 数据面设计

- 写入层负责消息持久化、分区路由和写入确认
- 分发层负责消费组协调、位点管理和重平衡

## 架构组成

### 核心组件

- Broker：承载消息写入、存储与分发
- Controller：管理主题、分区、副本和故障转移
- Client SDK：提供生产、消费和事务消息能力

### 关键交互

- 生产端将消息写入 Broker 并获得确认结果
- 消费组通过位点管理和重平衡机制获取分区消费权

## 部署形态

### 推荐拓扑

- 建议按业务域划分主题和分区，避免热点集中
- 跨地域场景可部署镜像复制或事件桥接通道

### 高可用策略

- 副本策略应覆盖单节点故障和单可用区故障
- 热点主题建议独立集群或独立资源池托管

## 快速入门

### 接入步骤

1. 创建主题与消费组。
2. 使用 SDK 发送业务事件。
3. 配置消费者重试、死信和位点提交策略。

### 首次验证

- 发送一条测试消息并确认消费者可成功消费
- 模拟消费失败，验证重试队列和死信队列是否符合预期

## 配置建议

### 基础配置

- 顺序消息与高吞吐消息应分开主题设计
- 重试次数和死信阈值应结合业务幂等能力设置

### 生产建议

- 分区数应按吞吐、并发和热点键分布综合规划
- 位点提交策略需与消费幂等和事务能力一起设计

## API 与 SDK

### 接口分层

- 提供主题管理、消息投递、消费位点和审计查询接口

### SDK 说明

- SDK 提供同步、异步、批量和事务消息模型
- 消费端需支持批量拉取、手动提交和异常回退策略

## 可观测性

### 关键指标

- 重点关注生产延迟、消费堆积、重试率和分区热点

### 运维重点

- 关注分区不均衡、消费堆积和副本同步延迟
- 关注重试风暴和死信积压趋势

## 典型场景

### 业务场景

- 订单事件驱动
- 异步削峰

### 平台场景

- 数据同步总线
- 跨系统事件桥接
