> 任务调度平台，负责定时任务编排、依赖调度、分片执行与运行治理。

## 产品定位

### 目标定位

Stellabe 提供面向平台级和业务级任务的统一调度能力，支持批处理、定时执行、工作流编排和失败重试。

### 适用边界

- 面向定时任务、工作流和批处理调度场景
- 面向任务回溯、补数和执行过程可视化
- 不替代流式计算引擎，而是聚焦任务编排和调度控制

## 核心能力

### 能力清单

- 支持 Cron 调度、工作流 DAG、任务依赖和分片执行
- 支持失败重试、幂等控制、补数和回溯执行
- 支持租户隔离、任务审计和运行态可视化

### 设计价值

- 将分散脚本和手工任务纳入统一治理体系
- 提高任务编排的可维护性和执行可靠性

## 概要设计

### 控制面设计

- 编排层负责工作流定义、依赖分析和任务发布
- 任务版本、审批和发布过程统一由控制面管理

### 数据面设计

- 调度层负责触发计算、分片分配和执行窗口控制
- 执行层负责 Worker 注册、状态回报和失败恢复

## 架构组成

### 核心组件

- Scheduler：触发任务并计算执行计划
- Executor：拉取任务、执行并回报结果
- Workflow Store：保存任务定义、运行记录和审计信息

### 关键交互

- Scheduler 依据触发条件生成执行计划并分派到 Executor
- Executor 上报执行状态和日志，形成完整运行记录

## 部署形态

### 推荐拓扑

- 建议控制面与执行面拆分部署，避免任务高峰冲击管理后台
- 对重任务场景建议按队列类型划分 Worker 集群

### 高可用策略

- 调度器需要保证单任务单次触发语义和抢占恢复能力
- 执行器建议按任务类型隔离资源池，防止互相影响

## 快速入门

### 接入步骤

1. 定义任务与执行器。
2. 配置 Cron 或 DAG 依赖关系。
3. 观察最近一次执行日志与重试记录。

### 首次验证

- 手动触发一次任务并确认执行器能正常拉取
- 模拟失败后验证重试和补偿链路是否生效

## 配置建议

### 基础配置

- 任务超时、重试次数和并发度应按任务类型独立配置
- 对补数任务建议设置时间窗口和资源上限

### 生产建议

- 关键任务需明确幂等策略和失败补偿逻辑
- DAG 任务建议限制深度和并发宽度，防止调度风暴

## API 与 SDK

### 接口分层

- 提供任务管理、执行控制、日志查询和工作流发布接口

### SDK 说明

- SDK 提供 Java/Go 执行器接入模板和心跳协议
- 执行器需显式回报进度、心跳和最终状态

## 可观测性

### 关键指标

- 重点关注触发延迟、成功率、重试深度和执行器利用率

### 运维重点

- 关注任务积压、执行器失联和重试雪崩
- 关注关键 DAG 节点失败对下游链路的传播影响

## 典型场景

### 业务场景

- 定时报表
- 数据补数

### 平台场景

- 发布后置任务编排
- 平台级工作流治理
