> 服务治理中枢，负责路由、负载、限流策略编排以及服务生命周期治理。

## 产品定位

### 目标定位

Stellorbit 为服务间调用提供统一治理策略，解决灰度发布、地域路由、版本隔离和故障切换等动态治理问题。

### 适用边界

- 面向服务调用链中的路由、重试、切流和故障隔离
- 面向多版本、多地域、多租户的治理规则编排
- 不直接承载注册和配置存储，而是消费外部契约信息

## 核心能力

### 能力清单

- 支持权重路由、标签路由、版本路由和地域路由
- 支持负载均衡、重试、超时和故障摘除编排
- 支持治理规则的灰度生效和回滚

### 设计价值

- 让流量治理从代码内嵌转为平台化编排
- 提升灰度发布和多活切流的可控性

## 概要设计

### 控制面设计

- 策略中心统一管理路由规则和治理模板
- 治理规则发布、灰度和回滚由控制面统一控制

### 数据面设计

- 执行面通过 Sidecar 或 SDK 在调用链路中实时生效
- 状态面负责熔断状态、实例画像和容量数据回传

## 架构组成

### 核心组件

- 规则仓库：存储路由和治理规则
- 执行引擎：在流量入口和调用客户端实施策略
- 状态同步器：回传实例健康、容量和异常事件

### 关键交互

- 客户端拉取或接收治理规则后在请求路径中即时执行
- 执行结果和实例状态回传至治理中心用于策略优化

## 部署形态

### 推荐拓扑

- 支持纯 SDK、Sidecar 和网关联动三种治理模式
- 核心控制面建议多副本部署，保证规则发布连续性

### 高可用策略

- 规则缓存需要具备本地持久化和断连容忍
- 网关与服务侧治理可组合部署，减少单点策略失效影响

## 快速入门

### 接入步骤

1. 为服务定义版本标签与流量分组。
2. 配置基础超时、重试和负载策略。
3. 发布灰度路由规则并观察治理指标。

### 首次验证

- 对一条灰度路由规则做小流量发布并核对命中情况
- 模拟实例异常，验证摘除和切流是否符合预期

## 配置建议

### 基础配置

- 路由规则建议分为全局策略、租户策略和服务级策略
- 重试与超时应联动配置，避免放大故障流量

### 生产建议

- 重要路由规则需具备审批和回滚预案
- 限制重试层级和超时叠加，避免请求雪崩

## API 与 SDK

### 接口分层

- 提供治理规则查询、灰度发布和状态回传接口

### SDK 说明

- SDK 暴露负载、路由、熔断和重试扩展点
- Sidecar 场景下建议将公共规则与业务规则分层管理

## 可观测性

### 关键指标

- 重点关注规则命中率、重试放大倍数、熔断次数和切流延迟

### 运维重点

- 关注异常规则扩散、切流延迟和回滚窗口
- 关注重试导致的依赖放大与容量失真

## 典型场景

### 业务场景

- 灰度发布
- 多活切流

### 平台场景

- 服务故障隔离
- 跨集群治理编排
