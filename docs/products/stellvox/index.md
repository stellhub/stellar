> 告警平台，负责告警规则、收敛降噪、通知编排和事件协同处置。

## 产品定位

### 目标定位

Stellvox 将指标、日志、链路和平台事件统一接入告警中心，实现统一触发、统一收敛和统一协同。

### 适用边界

- 面向指标、日志、链路和平台事件的统一告警治理
- 面向通知编排、值班协同和事件闭环处置
- 不替代监控数据源，而是作为告警决策与通知中枢

## 核心能力

### 能力清单

- 支持阈值告警、异常检测、事件告警和组合规则
- 支持告警分组、抑制、静默和升级策略
- 支持多渠道通知、值班编排和事件回溯

### 设计价值

- 将多来源异常统一收敛为可治理事件
- 降低告警风暴对值班和排障效率的冲击

## 概要设计

### 控制面设计

- 规则层负责策略配置、版本管理和灰度生效
- 通知模板、值班组和升级链路由控制面统一管理

### 数据面设计

- 引擎层负责实时计算、去重收敛和通知路由
- 协同层负责工单联动、值班轮值和处置追踪

## 架构组成

### 核心组件

- Alert Rule Engine：执行规则和收敛逻辑
- Notification Hub：对接邮件、IM、短信和 Webhook
- On-call Center：管理排班、升级和确认闭环

### 关键交互

- 规则引擎汇总多源事件并进行去重、抑制和分组
- 通知中心根据值班策略将事件投递到对应责任人

## 部署形态

### 推荐拓扑

- 建议事件计算与通知发送分离部署，提高告警高峰期韧性
- 多地域场景建议支持本地通知和全局汇总双通道

### 高可用策略

- 通知发送应具备失败重试、限速和多通道兜底能力
- 值班配置与规则配置需要支持版本化和快速回滚

## 快速入门

### 接入步骤

1. 接入指标、日志和 Trace 事件源。
2. 创建关键告警规则与通知策略。
3. 配置值班组和升级链路。

### 首次验证

- 触发一条测试告警，确认分组和通知链路已生效
- 验证确认、静默和升级流程是否符合预期

## 配置建议

### 基础配置

- 告警收敛窗口应按故障类型设置，避免风暴式通知
- 核心告警建议绑定 runbook 与责任团队

### 生产建议

- 规则粒度和通知频率需要平衡响应速度与噪声控制
- 高优先级告警建议配置多通道和升级机制

## API 与 SDK

### 接口分层

- 提供规则管理、通知模板、值班管理和事件查询接口

### SDK 说明

- SDK 支持业务自定义事件直报和回调处理
- 建议在业务侧显式声明事件级别、来源和责任标签

## 可观测性

### 关键指标

- 重点关注告警命中率、误报率、MTTA 和通知成功率

### 运维重点

- 关注通知延迟、值班规则漂移和升级链路失效
- 关注同一故障引发的告警风暴和误报比例

## 典型场景

### 业务场景

- 故障协同处置
- 值班轮值

### 平台场景

- 平台统一告警中心
- 多源事件收敛治理
