> 注册中心，负责服务实例注册、健康检查、发现订阅与拓扑变更推送。

## 产品定位

### 目标定位

Stellmap 面向微服务与基础设施组件提供统一注册发现能力，解决服务地址漂移、环境切换、实例伸缩和多集群路由等问题。

### 适用边界

- 面向服务注册、实例发现和订阅推送场景
- 面向多环境、多集群和灰度路由的服务目录管理
- 不承载业务配置和应用层流量治理逻辑

## 核心能力

### 能力清单

- 支持服务注册、实例摘除、心跳续约与健康检查
- 支持命名空间、集群、标签和版本维度的实例隔离
- 支持订阅推送与本地缓存，降低查询延迟

### 设计价值

- 通过注册发现统一服务入口，降低人工维护地址成本
- 通过标签和版本隔离支撑灰度发布与跨地域路由

## 概要设计

### 控制面设计

- 控制面负责服务元数据管理、注册协议适配和实例状态收敛
- 服务定义、分组标签和订阅关系统一由控制面维护

### 数据面设计

- 数据面负责高可用查询、增量推送和本地客户端缓存
- 一致性层采用多副本复制与租约模型保证实例信息可恢复

## 架构组成

### 核心组件

- 注册 API：接收实例注册、注销和续约请求
- 元数据存储：保存服务、标签和路由附加信息
- 订阅推送器：向客户端广播实例变更事件

### 关键交互

- 服务实例通过注册 API 上报状态并进行续约
- 调用方通过订阅通道或本地缓存获取最新实例集

## 部署形态

### 推荐拓扑

- 单 Region 三节点起步，跨可用区部署以降低单点故障
- 可在多 Region 模式下通过只读从集群承载本地发现流量

### 高可用策略

- 注册数据建议按租约和副本双重机制做故障恢复
- 客户端需启用本地缓存与短暂失联容忍策略

## 快速入门

### 接入步骤

1. 创建命名空间与服务标识。
2. 为应用接入 `stellmap-client` 并开启心跳续约。
3. 在调用方启用服务订阅与本地缓存。

### 首次验证

- 通过控制台或 API 查看实例是否已注册成功
- 模拟实例摘除并确认订阅方是否收到变更事件

## 配置建议

### 基础配置

- 生产环境建议区分注册超时、摘除阈值和推送批大小
- 对跨机房调用建议增加地域标签与就近路由策略

### 生产建议

- 注册超时和续约周期需要按服务启动时间与网络质量调优
- 推送批量与本地缓存刷新频率需要平衡实时性与吞吐

## API 与 SDK

### 接口分层

- 提供 HTTP/gRPC 注册接口和 Java、Go 客户端

### SDK 说明

- 客户端统一暴露注册、发现、订阅和事件监听能力
- 建议在启动阶段完成预热订阅，减少首次调用冷启动抖动

## 可观测性

### 关键指标

- 重点关注注册成功率、续约延迟、推送积压和缓存命中率

### 运维重点

- 关注实例频繁抖动、租约异常过期和推送通道积压
- 关注多集群间注册数据同步延迟

## 典型场景

### 业务场景

- 微服务实例发现
- 多集群灰度路由

### 平台场景

- 基础设施节点目录管理
- 统一服务目录治理
