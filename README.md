# 两仪

`两仪` 是一套自研微服务中间件体系，其正式英文总品牌为 `Stellar Axis`。这套体系以“星轴”作为工程表达，以“八卦”作为秩序骨架，以“灵觉层”作为 AI 扩展方向，面向高并发、高可用、可治理、可演进的分布式系统场景。

本仓库当前聚焦四类内容：

- 总品牌与体系定位
- 核心组件命名与职责映射
- 仓库布局与模块边界
- 依赖约束与架构决策

## 核心定位

两仪不是若干开源组件的简单拼装，而是一套统一命名、统一分层、统一依赖约束的中间件体系。整体设计遵循以下原则：

- 以 `Stellar Axis（两仪）` 作为唯一总品牌
- 以 `英文正式名 + 中文八卦名` 作为核心组件展示方式
- 以英文工程名作为仓库、模块、依赖坐标的唯一命名口径
- 以协议、SDK、事件和控制面作为跨产品协作边界

双栈定位如下：

- Java 框架：`Stellar Steel（两仪·刚）`
- Go 框架：`Stellar Titan（两仪·柔）`

## 核心组件命名释义

这一套命名不是功能直译，而是把工程角色与星轴意象、八卦义理绑定起来。英文名负责工程传播，中文名负责秩序映射，二者共同构成完整语义。

### StarMap · 乾仪

- 英文名 `StarMap` 指星图、星位坐标图。服务注册中心的本质，就是为所有服务实例提供统一坐标系，使调用方能够完成定位、发现与导航。
- 中文名 `乾仪` 取意于《易经》中“乾为天”。乾象征天道、首出庶物、统御全局，适合作为全局服务元数据中枢。
- “仪”一字兼有“仪象”“法仪”之意，表示它不是单个节点，而是一套被全系统共同遵循的定位尺度与秩序基准。

### Nebula · 坤仪

- 英文名 `Nebula` 指星云。星云既是承载物质的广阔背景，也是恒星孕育之所，契合配置中心“承载环境、孕育运行状态”的角色。
- 中文名 `坤仪` 取意于《易经》中“坤为地”。坤主承载、包容、生成，恰好对应配置中心承接所有服务运行参数、环境信息与动态规则。
- “仪”在这里强调它不是零散配置集合，而是系统级配置法度，是各服务得以稳定运行的共同底座。

### LightBeam · 离鉴

- 英文名 `LightBeam` 指光束。链路追踪的作用是把原本不可见的调用路径照亮，使路径、耗时、异常与因果关系清晰可见。
- 中文名 `离鉴` 取意于《易经》中“离为火”“离为日”“离为明”。离卦主光明、照见、附丽，最适合表达可观测性与链路显影。
- “鉴”有镜鉴、明鉴之意，强调系统不仅被照亮，还能借此审视自身、定位问题、反观运行状态。

### Orbit · 巽策

- 英文名 `Orbit` 指轨道。治理系统本质上是在定义请求如何沿着既定轨道运行，包括路由、负载、灰度、流量牵引与治理策略。
- 中文名 `巽策` 取意于《易经》中“巽为风”。风无孔不入、因势而行，正对应服务治理对流量的柔性调度与细致渗透。
- “策”表示策略、调度与筹划，强调治理不是强制拦截，而是通过规则设计让流量沿正确路径运行。

### Pulsar · 艮闸

- 英文名 `Pulsar` 指脉冲星。脉冲星的特征是周期稳定、节律精准，这与限流、配额、熔断中的阈值控制和频率控制高度契合。
- 中文名 `艮闸` 取意于《易经》中“艮为山”。艮主止，表示在流量失控、故障蔓延或压力突增时，系统必须能及时止住洪峰。
- “闸”强调它不是抽象的“止”，而是工程上明确的闸门机制，用于限流、隔离、熔断和保护下游。

### Chronos · 震策

- 英文名 `Chronos` 来自时间秩序的意象，天然适合表达调度系统对于时钟、节拍、周期与执行时点的掌控。
- 中文名 `震策` 取意于《易经》中“震为雷”。雷主发动、惊起、启行，适合作为定时任务和调度触发的象征。
- “策”表示调度编排与执行策略，说明它不仅负责“何时响”，更负责“如何触发、如何协调、如何分发”。

### Singularity · 坎锁

- 英文名 `Singularity` 指奇点，强调在复杂分布式空间中收束为唯一控制点，这与锁的排他性与唯一性高度一致。
- 中文名 `坎锁` 取意于《易经》中“坎为水”“坎为险”。坎象征险阻、陷阱与边界，契合并发竞争中“进入即受约束”的控制语义。
- “锁”明确指出其工程职责，即通过排他控制把无序竞争收束为可管理、可串行的访问秩序。

### EventHorizon · 兑门

- 英文名 `EventHorizon` 指事件视界，是内外边界的临界面。网关正是系统内部与外部流量的边界层，承担接入、聚合、鉴权与转发职责。
- 中文名 `兑门` 取意于《易经》中“兑为泽”“兑为口”。兑象征开口、表达与出入之门，天然适合作为系统门户的命名。
- “门”强调它是边界入口，是所有外部请求进入体系前必须经过的统一关口。

### TaiJi Flow · 太极流

- 英文名 `TaiJi Flow` 直接强调“流”，对应消息、事件、异步数据和系统内能量的流转。
- 中文名 `太极流` 取意于“易有太极，是生两仪”。太极代表体系中心的统一源头，而消息流正是把各组件连接起来的中枢性流动介质。
- 之所以不再采用更分散的命名，是因为消息系统在体系中承担的是“中心流转”角色，而不是一个普通边缘组件。

### Constellation · 观衡

- 英文名 `Constellation` 指星座。指标系统的本质不是单个点，而是把大量离散时序指标组织成有结构的可观察图谱。
- 中文名 `观衡` 中，“观”可对应《易经》观卦之“观国之光”，强调观测、审视与洞察；“衡”表示度量、权衡、标尺。
- 这套名字非常适合承载以 `Grafana + Prometheus + OpenTelemetry` 为底层的统一指标与观测平台。

### Beacon · 警烽

- 英文名 `Beacon` 指信标、灯塔、告警信号源，天然适合表示告警、通知、升级与值守联动。
- 中文名 `警烽` 取意于古代“烽火传警”，强调告警平台的本质是快速示警、逐级传警与全局联动。
- 这个名字适合统一承载规则告警、阈值告警、值班路由、通知升级和事件编排。

### Aegis · 御信

- 英文名 `Aegis` 表示守护之盾，适合作为统一鉴权与零信任平台的工程品牌。
- 中文名 `御信` 的重点不在“相信”，而在“御之有度、信之有据”，表示所有身份与访问都必须经过持续验证和策略裁定。
- 这个名字适合承载统一身份、认证、授权、策略判定、服务间信任与零信任访问控制。

### Keystone · 金匮

- 英文名 `Keystone` 既有“关键石”之意，也能自然联想到 key 的核心地位，适合作为 KMS 与密钥分发系统的工程名称。
- 中文名 `金匮` 取意于古代秘匣、密藏之柜，带有“珍重收藏、严密封存”的意味。
- 这个名字适合承载密钥生命周期管理、证书分发、机密轮换、加解密材料托管等能力。

## AI 命名释义

### Spirit Axis · 灵觉层

- 英文名 `Spirit Axis` 表示认知与智能能力的中轴线。
- 中文名 `灵觉层` 强调它不是一个单点模型，而是一层负责理解、记忆、推理与执行的认知结构。

### ZhongFu Engine · 中孚

- `中孚` 出自《易经》六十一卦，核心意涵是真诚、内信、感而能通。
- 作为推理引擎，这个名字强调模型不是机械匹配，而是对语义与意图进行可信理解与内在贯通。

### DaChu Memory · 大畜

- `大畜` 出自《易经》二十六卦，有积蓄、涵养、储备之义。
- 记忆层的本质正是对知识、上下文与经验进行持续蓄积，并在需要时稳定释放。

### Sui Agent · 随位

- “随”取自《易经》随卦，强调因时而动、随势而行。
- “位”强调代理的角色不是无边界行动，而是在明确职责位势中的自适应执行。

### Xian MCP · 咸

- `咸` 出自《易经》咸卦，主感应、交感。
- 它用于上下文感知协议，表示 AI 与外部资源之间首先发生的是感知与感应。

### Heng MCP · 恒

- `恒` 出自《易经》恒卦，主常、主久、主持续。
- 它用于工具执行协议，强调工具调用必须稳定、可重复、可持续，不是偶发性的动作。

### Meng MCP · 蒙

- `蒙` 出自《易经》蒙卦，主启蒙、导引、开示。
- 它用于 Prompt 引导协议，表示系统通过结构化引导帮助模型进入正确理解与执行路径。

## 网关家族命名建议

你现在有内部网关、外部网关、LLM 网关、边缘网关。它们不适合使用完全相同的完整名称，但非常适合共享同一个网关家族主名。

建议规则：

- 家族主名统一使用 `EventHorizon · 兑门`
- 不同项目通过二级限定词区分
- 仓库名、模块名和部署单元继续使用英文工程后缀区分

推荐方案如下：

| 场景 | 英文建议 | 中文建议 | 说明 |
| :--- | :--- | :--- | :--- |
| 内部网关 | `EventHorizon Internal` | `兑门·内关` | 面向服务网格内部、东西向流量与内部路由治理 |
| 外部网关 | `EventHorizon External` | `兑门·外关` | 面向北南向入口、开放接入与统一暴露 |
| LLM 网关 | `EventHorizon LLM` | `兑门·灵关` | 面向模型路由、模型鉴权、Token 治理与 AI 请求编排 |
| 边缘网关 | `EventHorizon Edge` | `兑门·边关` | 面向边缘节点、接入加速、就近转发与边缘计算场景 |

结论：

- 它们适合共用同一个家族根名 `EventHorizon / 兑门`
- 它们不适合使用完全相同的完整项目名
- 最终应采用“同一家族名 + 明确场景后缀”的方式命名

如果落实到仓库名，建议分别使用：

- `event-horizon-internal`
- `event-horizon-external`
- `event-horizon-llm`
- `event-horizon-edge`

整个体系采用“品牌聚合仓库 + 核心产品仓库 + 配套能力仓库”的三层结构。

顶层聚合仓库：

- `stellar-axis`
- `stellar-steel`
- `stellar-titan`
- `spirit-axis`
- `stellar-control-plane`
- `stellarctl`

- `starmap`
- `nebula`
- `lightbeam`
- `orbit`
- `pulsar`
- `chronos`
- `singularity`
- `event-horizon`

```text
{product}/
├── docs/
├── api/
├── server/
├── clients/
├── starters/
├── sidecars/
├── operators/
├── deploy/
├── test/
└── examples/
```

## 依赖规则

整套体系统一采用单向依赖约束。

允许的原则：

- 依赖公开 API、SDK 和稳定协议
- 通过控制面、事件流和官方客户端实现跨产品协作
- `starter` 依赖 `clients/api`
- `sidecar` 依赖 `clients/api/runtime`
- `operator` 依赖 `api` 和 Kubernetes Runtime

禁止的原则：

- `starter -> server`
- `client -> server`
- `operator -> server`
- `control-plane -> product/internal`
- `productA/server -> productB/server`
- `examples -> product/internal`
